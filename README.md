# RoutineFlow — Backend

A backend for a routine/habit-tracking app. Users set **goals**, break them into **tasks**, attach a **plan** (weekly, monthly, or a custom date range), and log daily progress. Each day they can also record an **emotion** (with emoji) and a diary entry. Progress is aggregated into weekly/monthly calendar views, and users get a weekly summary notification of how they did.

The idea: help people actually *feel* their progress — a goal still counts as a win for the day even if you only did one of ten tasks under it — and give them a simple emotional log alongside it.

## Why this project exists

I'm building this as a portfolio backend to demonstrate real engineering fundamentals end-to-end: relational schema design, JWT-based auth done properly (hashed refresh tokens, rotation, revocation), background jobs with real scheduling/idempotency concerns, and deliberate handling of an easy-to-get-wrong edge case (a soft-delete + timed data-retention policy with staged user warnings). I'm using it to show recruiters and hiring teams — including for backend/API roles built around PostgreSQL, clean service layering, and a quality-first engineering approach — that I can take a set of fuzzy product requirements, ask the right clarifying questions, and turn them into a working, tested system rather than just CRUD scaffolding.

The stack here is C#/ASP.NET Core, but the underlying skills (schema design, API design, auth, background jobs, migrations, thinking through edge cases before they hit production) are the same ones that matter on any modern backend stack, including TypeScript/Next.js/serverless-style roles.

## Tech Stack

- **C# / ASP.NET Core 8** Web API, MVC controllers
- **PostgreSQL** via **EF Core** + Npgsql, code-first migrations
- **JWT** authentication (short-lived access token + rotating, hashed refresh tokens)
- **BCrypt** password hashing
- Layered architecture: `Controllers` → `Services` → `Repositories` → `Data` (EF Core `DbContext` + Fluent API configurations)
- Background jobs (`BackgroundService`) for scheduled work (weekly summaries, retention-policy sweeps)
- **FluentValidation** for request validation, surfaced as RFC 9110 `ProblemDetails`/`ValidationProblemDetails` from a global exception middleware and action filter
- Per-IP rate limiting on login to slow down credential-stuffing attempts
- **Serilog** structured logging (console + rolling file sink), with request logging middleware
- Integration tests (`WebApplicationFactory` + **Testcontainers** for a real, disposable PostgreSQL instance per test run) covering auth, ownership isolation, and full end-to-end flows
- Swagger / OpenAPI (with JWT bearer auth wired into the UI) for interactive API exploration

## Features

- **Auth**: register, login, refresh (rotating refresh tokens, reuse detection), logout
- **Goals & Tasks**: CRUD, tasks nested under goals
- **Plans**: attach a Weekly / Monthly / Custom(date-range) schedule to a goal
- **Daily tracking**: mark a task done for a day (increments a count, accepts an optional note); a goal counts as "achieved" for the day if *any* of its tasks were completed
- **Daily reports**: log an emotion (with emoji) and/or a diary entry per day
- **Calendar views**: weekly and monthly aggregation of emotion + per-goal achievement
- **Notifications**: in-app weekly progress summaries, generated by a scheduled background job
- **Goal retention policy**: deleting a goal soft-deletes it; it's recoverable for 6 months, with warning notifications sent at the 1-month and 1-week marks before permanent deletion
- **Consistent error responses**: every error (validation failure, not-found, conflict, unexpected exception) comes back as a standard `ProblemDetails` payload

## Project Structure

```
Controllers/     API endpoints (Auth, Goals, Tasks, Plans, Calendar, Notifications, ...)
Services/        Business logic
Repositories/    EF Core data access, one per entity
Models/
  Entities/      EF Core entity classes
  Enums/         PlanType, EmotionType, NotificationType
DTOs/            Request/response contracts + FluentValidation validators, grouped by feature

  ApplicationDbContext.cs

```

## Getting Started

### Prerequisites

- [.NET 8 SDK](https://dotnet.microsoft.com/download)
- PostgreSQL (locally, or via Docker — see below)

### Run PostgreSQL (Docker)

```bash
docker run -d --name routineflow-pg -e POSTGRES_PASSWORD=postgres -e POSTGRES_DB=routineflow -p 5432:5432 postgres:16
```

### Configure secrets

Connection string and JWT signing key are kept out of source control via [user-secrets](https://learn.microsoft.com/aspnet/core/security/app-secrets):

```bash
dotnet user-secrets init
dotnet user-secrets set "ConnectionStrings:Default" "Host=localhost;Port=5432;Database=routineflow;Username=postgres;Password=postgres"
dotnet user-secrets set "Jwt:SigningKey" "<a long random base64 string>"
```

### Apply migrations & run

```bash
dotnet tool install --global dotnet-ef
dotnet ef database update
dotnet run
```

Then open `https://localhost:<port>/swagger` for the interactive API docs — use the **Authorize** button to paste in a JWT access token after logging in.

### Run the tests

The integration test suite spins up its own disposable PostgreSQL container via [Testcontainers](https://testcontainers.com/) — no manual database setup needed, just a running Docker daemon:

```bash
dotnet test
```
